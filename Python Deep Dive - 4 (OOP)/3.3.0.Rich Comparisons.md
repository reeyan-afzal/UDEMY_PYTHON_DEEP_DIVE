This is quite straightforward. We can choose to implement any number of these rich comparison operators in our classes.

Furthermore, if one comparison does not exist, Python will try to reverse the operands and the operator (and unlike the arithmetic operators, both operands can be of the same type).

Let's use a 2Dô `Vector`ô class to check this out:

By default, Python will useô `is`ô when we do not provide an implementation forô `==`. In this case we have two different objects, so they do not compareô `==`.

Let's change that:

class Vector:

Let's implement theô `<`ô operator:

We'll consider a Vector to be less than another vector if its length (Euclidean) is less than the other.

We're going to make use of theô `abs`ô function for this, so we'll define theô `__abs__`ô method as well.

```python
from math import sqrt

class Vector:
    def __init__(self, x, y):
        self.x = x
        self.y = y
        
    def __repr__(self):
        return f'Vector(x={self.x}, y={self.y})'
        
    def __eq__(self, other):
        if isinstance(other, tuple):
            other = Vector(*other)
        if isinstance(other, Vector):
            return self.x == other.x and self.y == other.y
        return NotImplemented
    
    def __abs__(self):
        return sqrt(self.x ** 2 + self.y ** 2)
    
    def __lt__(self, other):
        if isinstance(other, tuple):
            other = Vector(*other)
        if isinstance(other, Vector):
            return abs(self) < abs(other)
        return NotImplemented
```

v2 = Vector(1, 1)

How did that work? Well Python could not find aô `__ne__`ô method, so it delegated toô `__eq__`ô instead:

We can easily see this by adding a print statement to ourô `__eq__`ô method:

```python
from math import sqrt

class Vector:
    def __init__(self, x, y):
        self.x = x
        self.y = y
        
    def __repr__(self):
        return f'Vector(x={self.x}, y={self.y})'
        
    def __eq__(self, other):
        print('__eq__ called...')
        if isinstance(other, tuple):
            other = Vector(*other)
        if isinstance(other, Vector):
            return self.x == other.x and self.y == other.y
        return NotImplemented
    
    def __abs__(self):
        return sqrt(self.x ** 2 + self.y ** 2)
    
    def __lt__(self, other):
        if isinstance(other, tuple):
            other = Vector(*other)
        if isinstance(other, Vector):
            return abs(self) < abs(other)

```

v2 = Vector(1, 1)
```

```

In many cases, we can derive most of the rich comparisons from just two base ones: theô `__eq__`ô and one other one, maybeô `__lt__`, orô `__le__`, etc.

For example, ifô `==`ô andô `<`ô is defined, then:

- `a <= b`ô isô `a == b or a < b`
- `a > b`ô isô `b < a`
- `a >= b`ô isô `a == b or b < a`
- `a != b`ô isô `not(a == b)`

On the other hand if we defineô `==`ô andô `<=`, then:

- `a < b`ô isô `a <= b and not(a == b)`
- `a >= b`ô isô `b <= a`
- `a > b`ô isô `b <= a and not(b == a)`
- `a != b`ô isô `not(a == b)`

So, instead of defining all the various methods, we can use theô `@total_ordering`ô decorator in theô `functools`ô module, which will work withô `__eq__`ô andô **one**ô other rich comparison method, filling in all the gaps for us:

```python
from functools import total_ordering

@total_ordering
class Number:
    def __init__(self, x):
        self.x = x
        
    def __eq__(self, other):
        print('__eq__ called...')
        if isinstance(other, Number):
            return self.x == other.x
        return NotImplemented
    
    def __lt__(self, other):
        print('__lt__ called...')
        if isinstance(other, Number):
            return self.x < other.x
        return NotImplemented
```

```python
a = Number(1)
b = Number(2)
c = Number(1)
```

```python
a < b
```

```python
a <= b
```

You'll notice thatô `__eq__`ô was not called - that's becauseô `a < b`ô was True, and short-circuit evaluation. In this next example, though, you'll see both methods are called:

```python
a <= c
```

One thing I want to point out, according to the documentation theô `__eq__`ô is not actuallyô **required**. That's because as we saw earlier, all objects have aô **default**ô implementation forô `==`ô based on the memory address. That's usually not what we want, so we normally end up defining a customô `__eq__`ô implementation as well.
